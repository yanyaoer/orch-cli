Coding agents as supervised, auditable local jobs.

把编码 Agent 跑成可监督、可审计的本地任务。

orch dispatches headless codex / claude / pi / omp workers through a signed local mail bus, claims work atomically, and validates every result against a role schema. No daemon — each command reads local files, advances one step, and exits.

orch 通过签名的本地 mail bus 派发 headless 的 codex / claude / pi / omp worker， 原子 claim 任务，并按角色 schema 校验每一个结果。没有 daemon——每个命令读本地文件、 推进一步、退出。

Daemonless

无 daemon

Every command is a stateless reconciler over local files. The per-run supervisor lives exactly as long as one run — process-group kill, timeout, heartbeat, honest terminal states.

每个命令都是基于本地文件的无状态 reconciler。supervisor 与单个 run 同生共死——进程组 kill、超时、心跳、诚实的终止状态。

Auditable by files

文件即审计

Signed events, verified mail-events.jsonl, per-event claim leases, immutable spec.json, append-only events.jsonl. Swap the controller, keep the state.

签名事件、已验证的 mail-events.jsonl、逐事件 claim 租约、不可变 spec.json、只追加的 events.jsonl。换主控不丢状态。

Schema-gated results

结果过 schema 门

Workers must return orch.result/<role>/v1 JSON. Benign deviations are coerced, failed extraction keeps the raw answer as result.raw.md, and silent zero-output exits fail the run.

worker 必须返回 orch.result/<role>/v1 JSON。无害偏差自动归一，提取失败时原始回答保留为 result.raw.md，零输出的静默退出按失败处理。

01 — ROLES

Permissions and models match the role.

权限与模型随角色匹配。

The reviewer role launches every provider read-only; write roles and the verifier keep write-capable access. The claude driver also picks model and effort per role.

reviewer 角色下所有 provider 都以只读方式启动；写角色与 verifier 保留写权限。claude driver 还会按角色选择模型与推理力度。

An explicit --model <ref> is recorded in spec.json and overrides the provider model for model-aware drivers.

显式 --model <ref> 会记录进 spec.json，并对支持模型选择的 driver 覆盖默认模型。

02 — OBSERVE

One run directory tells the whole story.

一个 run 目录讲完整个故事。

Lifecycle authority stays in events.jsonl and status.json. The provider-native stream is kept verbatim in native.jsonl — and events tail --native normalizes it into provider-independent progress events, so you watch every worker the same way.

生命周期权威在 events.jsonl 与 status.json。provider 原生流原样保留在 native.jsonl——events tail --native 把它规范化为与 provider 无关的 进度事件，用同一种方式观察任何 worker。

$XDG_STATE_HOME/orch/<repo_key>/mrs/<mr>/runs/<run_id>/
├── spec.json      immutable input snapshot
├── status.json    state · pid · shas · resume id
├── events.jsonl   orch lifecycle — the authority
├── native.jsonl   provider stream, verbatim
├── result.json    schema-validated role result
├── stdout.log · stderr.log
└── artifacts/
$ orch events tail --run <run_id> --native

03 — QUICKSTART

One command fans a task across agents.

一条命令把任务分发给多个 agent。

Install with one curl line (release binary; upgrade later with orch update), register default mail agents once, then publish, claim, and run ride the local mail bus in one shot. The manual submit / route / claim pipeline stays available for fine control.

一条 curl 命令安装 release binary（之后用 orch update 升级），注册一次默认 mail agents，之后发布、claim、运行一条命令走完本地 mail bus。手动 submit / route / claim 流水线仍然可用。 

# install the latest release binary into ~/.local/bin
$ curl -fsSL https://raw.githubusercontent.com/yanyaoer/orch-cli/main/install.sh | sh
$ orch update                                 # later: self-update to the latest release

$ orch mail agent defaults                    # once
$ orch workspace add --id my-repo --path .

# one-shot fan-out: publish + claim + run per agent
$ orch cross-review --thread review-123 --task review.md
$ orch fanout --thread verify-1 --role verifier --task verify.md
$ orch investigate --thread research-1 --task question.md

# follow the runs (mr == thread)
$ orch status --mr review-123
$ orch result --run <run_id> --wait
$ orch events tail --run <run_id> --native -n 20

# single supervised run; --mr resolves from the task or branch
$ orch run create --role reviewer --agent codex --task review.md

# record the verdict; mirror to PR/MR only with --execute
$ orch decision accept --run <run_id> --reason "reviewed"
$ orch mirror sync --mr 123 --execute

04 — STATUS

What ships now — and what doesn't.

已实现什么，边界在哪。

Commands

命令

  • bare orch (overview) · verdict --wait · wait
  • run create · run list · run reap · status
  • cross-review · fanout · investigate
  • mail submit · mail route · mail claim · mail reply result
  • events tail [--native] · result --wait
  • decision · mirror · mirror sync · workspace
  • chatgpt-bridge · handoff-pro

Boundaries

边界

  • No long-running daemon; stale runs are flagged on read and reaped explicitly.
  • The shipped message bus is local Maildir state, not a multi-machine database.
  • Mirroring is outbox-first: nothing touches the network without --execute.
  • Provider sessions never auto-resume "latest"; exact resume needs an explicit session id.
  • chatgpt-bridge is a read-only Cloudflare Worker MCP bridge.
  • 没有常驻 daemon；僵尸 run 读取时标记 stale，需显式 reap。
  • 当前消息总线是本地 Maildir 状态，不是多机数据库。
  • PR/MR 镜像 outbox 先行：没有 --execute 不碰网络。
  • provider 会话从不自动续"最近一次"；精确恢复必须显式给 session id。
  • chatgpt-bridge 是只读的 Cloudflare Worker MCP 桥。
